# 🚢 MemoryLane Backend Deployment Guide

This guide provides step-by-step instructions to deploy the FastAPI application gateway and Redis background workers to production using **Railway**, connected to **Supabase (PostgreSQL)**, **AWS S3**, **Razorpay**, and external notification services.

---

## 📋 Infrastructure Requirements

To run MemoryLane in production, you will need active accounts with:
1. **Railway**: Hosts the FastAPI web dyno and Redis queue.
2. **Supabase**: Hosts the PostgreSQL database, Authentication provider, and Realtime notification client.
3. **AWS (S3 & IAM)**: Houses high-resolution image uploads.
4. **Razorpay**: E-commerce payment collections.
5. **Resend**: Transactional emails.
6. **Twilio**: Transactional WhatsApp alerts.

## 🗄️ Step 1: Database & Storage Configuration

### A. AWS S3 Bucket Setup & CORS Policy
The frontend uploads high-resolution images directly to S3 via presigned URLs generated by the backend. 
1. Create an S3 Bucket (e.g. `memorylane-photos`).
2. Go to **Permissions** -> **Cross-origin resource sharing (CORS)** and paste the following configuration:
```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["PUT", "POST", "GET", "HEAD"],
    "AllowedOrigins": ["https://your-frontend-domain.vercel.app", "http://localhost:3000"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```
3. Create an AWS IAM user with programmatic access credentials. Attach the following custom inline policy to restrict access to your bucket:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::your_bucket_name",
        "arn:aws:s3:::your_bucket_name/*"
      ]
    }
  ]
}
```

### B. Supabase PostgreSQL Schema & Realtime
1. Open the Supabase Database SQL Editor.
2. Paste and run the [schema.sql](file:///c:/Users/aryan/OneDrive/Desktop/Up%20Stack/ML/memorylane-backend/schema.sql) scripts to establish the database schema, foreign key relations, RLS profiles, and trigger routines.
3. Enable **Supabase Realtime** replication for tracking processing logs instantly in the browser:
```sql
ALTER PUBLICATION supabase_realtime ADD TABLE public.orders;
ALTER PUBLICATION supabase_realtime ADD TABLE public.photo_batches;
```

## 🚄 Step 2: Railway Project Setup

Railway reads the project [Procfile](file:///c:/Users/aryan/OneDrive/Desktop/Up%20Stack/ML/memorylane-backend/Procfile) and spins up two services:
1. **API Web Server**: Running `uvicorn main:app --host 0.0.0.0 --port $PORT`
2. **Background AI Pipeline Worker**: Running `python worker.py`

### Deploying the Backend on Railway
1. Log in to [Railway](https://railway.app).
2. Click **New Project** > **Deploy from GitHub repository** > select `memorylane-backend`.
3. Inside the project dashboard, click **New** > **Database** > **Redis** to provision a Redis instance. Railway automatically injects the `REDIS_URL` connection parameter into services in the same project.
4. Select the `memorylane-backend` service, navigate to **Settings**, and click **Generate Domain** under the Public Networking section. Copy this URL (e.g. `https://your-backend.up.railway.app`) as your API root.

## ⚙️ Step 3: Production Environment Variables

Configure these keys under the **Variables** tab of the `memorylane-backend` service on Railway:

| Variable | Description | Example / Recommendations |
| :--- | :--- | :--- |
| `ENV` | Application environment status | `production` (enforces strict CORS rules) |
| `PORT` | Web server port (Railway default) | `8000` |
| `FRONTEND_URL` | Production web application address | `https://memorylane.vercel.app` (no trailing slash) |
| `SUPABASE_URL` | Supabase API connection URL | `https://[project-id].supabase.co` |
| `SUPABASE_ANON_KEY` | Supabase anon key | Retrieve from settings -> API |
| `SUPABASE_SERVICE_KEY` | Service role key (Secret) | Bypasses RLS to write curation results |
| `AWS_ACCESS_KEY_ID` | Programmatic IAM access key | Created in S3 step above |
| `AWS_SECRET_ACCESS_KEY` | Programmatic IAM secret key | Created in S3 step above |
| `AWS_REGION` | S3 bucket region code | `ap-south-1` (Mumbai) |
| `S3_BUCKET_NAME` | Dedicated S3 bucket name | `memorylane-photos` |
| `GEMINI_API_KEY` | Google AI Studio Key | Used for primary image description tasks |
| `ANTHROPIC_API_KEY` | Claude API key | Fallback option for text sequencing |
| `RAZORPAY_KEY_ID` | Razorpay Merchant key | Format: `rzp_live_...` or `rzp_test_...` |
| `RAZORPAY_KEY_SECRET` | Razorpay Merchant secret key | Kept private |
| `RESEND_API_KEY` | Resend API client key | Format: `re_...` |
| `TWILIO_ACCOUNT_SID` | Twilio Account identifier | Format: `AC...` |
| `TWILIO_AUTH_TOKEN` | Twilio client credentials secret | Keep secure |
| `TWILIO_WHATSAPP_FROM` | Registered WhatsApp Sender number | `whatsapp:+14155238886` |
| `SENTRY_DSN` | Sentry logging URL (optional) | `https://sentry.io/...` |

## 📈 Step 4: Health Checking & Post-Deploy Verification

### 1. Check Service Health
Visit your generated Railway backend URL at the `/health` path (e.g. `https://your-backend.up.railway.app/health`). The API should return:
```json
{"status": "ok", "service": "memorylane-api"}
```

### 2. Verify Background Logs
1. Navigate to the **Worker** service in Railway.
2. Open the **Deployments / Logs** tab. You should see connection output confirming that the worker is listening for jobs on the Redis broker:
```text
2026-07-01 00:30:15 - worker - INFO - Starting background gallery compilation task queue listener...
```

### 3. Connect the Frontend
Provide the frontend service deployment (Vercel) with the backend Railway domain as `NEXT_PUBLIC_API_URL`.
